02零基础背景包

学过但生疏？下面 5 个概念用一句话唤醒，不用从零讲。

背景补充 ① · 程序 vs 进程（唤醒）
程序是硬盘上的 .exe 文件，死的；进程是你双击它之后，在内存里跑起来的活的实例。一个程序可以同时跑多个进程（开 3 个浏览器窗口 = 3 个进程）。
一句话记忆：程序是菜谱，进程是正在被厨师做的这道菜。
背景补充 ② · 并发 vs 并行（唤醒）
并行=多个 CPU 真的同时干（多核）；并发=一个 CPU 快速切换，让你"感觉"同时在跑。本章默认讨论单 CPU 并发。
一句话记忆：并发是一个人同时炒两个菜来回颠勺，并行是两个人各炒各的。
背景补充 ③ · 临界资源 / 临界区（唤醒）
临界资源=一次只能一个人用的东西（打印机、共享变量）；临界区=你代码里访问这个资源的那几行。
一句话记忆：临界资源是公厕，临界区是你进厕所到出来的那段时间。
背景补充 ④ · 信号量 PV（唤醒）
信号量 S 是个计数器。S=3 表示"还有 3 个名额"；S=-2 表示"2 人在排队等"。P 操作=拿一个名额（S-1，拿不到就排队等）；V 操作=还一个名额（S+1，有人在等就叫他）。
一句话记忆：P = Pass / 等（拿走名额），V = Vacate / 放（释放名额）。
背景补充 ⑤ · 死锁（唤醒）
死锁=几个进程互相拿着对方想要的资源，谁都不肯先放，全部卡死。
一句话记忆：过独木桥的两个人，都不肯退。

03原文问题定位

表面问题 这章在讲什么？
进程管理的四大板块：进程基本概念、处理机调度、进程同步（PV）、死锁处理。
深层问题 它在回答什么？
单个 CPU，多个进程都想跑——OS 怎么管？分三个子问题：
① 谁先跑？跑多久？（调度算法）
② 怎么防止它们互相踩共享数据？（同步与互斥）
③ 怎么防止它们卡死谁也动不了？（死锁）
只记一个问题 「多个进程争抢有限资源，OS 如何维持秩序？」
调度管 CPU 时间分配，同步管共享数据访问，死锁管资源分配安全性——三条线都指向"资源竞争下的秩序"。
事实原文：「进程是资源分配和独立运行的基本单位」「死锁：两个或以上进程因竞争资源互相等待的僵局」

05核心判断与依据

事实
判断 1：进程是资源分配和独立运行的基本单位
程序是静态的，进程是动态的；进程能独立运行、能与其他进程并发、是竞争资源的基本单位。
原文依据：「进程=程序段+数据集合+PCB」「进程是竞争计算机系统有限资源的基本单位」
判断
判断 2：调度的核心是权衡"用户满意"和"系统效率"
面向用户的准则（周转时间短、响应快、截止时间保证、优先权）vs 面向系统的准则（吞吐量高、CPU 利用率好、资源平衡）。
原文依据：「调度基本准则分面向用户、面向系统两组」
判断
判断 3：PV 操作中 P 的顺序不能颠倒，V 的顺序可以任意
先 P 互斥信号量再 P 同步信号量会死锁（互斥锁住了却等不到同步）；V 操作只释放不阻塞，所以顺序无所谓。
原文依据：「P 的顺序不能颠倒，V 的顺序任意」
这是大题代码题最容易扣分的点。
判断
判断 4：死锁的四个必要条件必须同时成立
互斥、不剥夺（请求和保持）、部分分配、环路等待——缺一不可。预防死锁就是破坏其中之一。
原文依据：「产生死锁的四个必要条件：互斥、不剥夺、部分分配、环路等待」

06关键概念拆解

PCB（进程控制块）
原文定义：描述进程的数据结构，是进程存在的唯一标志。
人话：OS 给每个进程发的"身份证"，记录它叫什么、在哪儿、状态如何、优先级多少、现场寄存器值。进程入睡时，断点（CPU 现场信息）就存在 PCB 里。
关键：系统有 n 个进程，就绪队列最多 n-1 个（另一个在运行）。
不要误解成：PCB 不是程序的代码段，是"管理信息"的集合。
进程三态转换
就绪（万事俱备只欠 CPU）→ 运行（被调度选中）→ 等待（等 I/O 或资源）。
关键转换：
• 就绪→运行：被调度
• 运行→就绪：时间片到
• 运行→等待：I/O 请求 / P 操作阻塞
• 等待→就绪：I/O 完成 / V 操作唤醒
注意：没有"就绪→等待"和"等待→运行"的直接转换！这是高频选择考点。
线程（Thread）
原文：进程内被独立调度的最小单位，又称轻权进程（LWP）。
关键区别：传统 OS 中进程既是资源分配单位又是调度单位；现代 OS 中进程只是资源分配单位，线程才是调度单位。一个进程可以有多个线程，共享进程的资源。
原文依据：「资源分配给进程，处理机分给线程」
响应比 R（HRN 算法核心）
R = 1 + 已等待时间 / 需运行时间
直觉：等得越久 R 越大（偏向 FCFS），运行时间越短 R 越大（偏向 SJF），所以 HRN 是两者的折中。
原文依据：「最高响应比作业优先算法是对 FCFS 和 SJF 的综合平衡」
银行家算法的安全状态
安全状态=存在一个进程序列 <P1..Pn>，按这个顺序分配资源，每个进程都能拿到所需资源跑完并归还。
关键：安全状态 → 一定不会死锁；不安全状态 → 不一定死锁（只是"可能"死锁）。这是高频误解点。
原文依据：「并非所有不安全状态都是死锁状态……反之只要系统处于安全状态，系统便可避免进入死锁状态」

08手把手计算演示

本章大题的三种计算题型，每种给你一道完整例题，从读题到答案每一步都写出来。看懂这三道，大题就稳了一半。

演示 1 · 进程调度参数计算（FCFS + SJF）

这是最典型的大题：给 3~4 个进程，要你列表、算周转时间、算平均带权周转时间。

已知：单道批处理系统，采用短作业优先（SJF）。
进程 P1/P2/P3 依次到达：
P1：到达 9:00，需运行 60 分钟
P2：到达 9:10，需运行 45 分钟
P3：到达 9:15，需运行 25 分钟
求：每个进程的开始时间、完成时间、周转时间，以及平均周转时间。
第 1 步 · 理解 SJF 的调度规则
SJF = 短作业优先。但不是全局找最短，而是：每当 CPU 空闲时，从"已经到达的"进程里挑运行时间最短的。
关键：9:00 时只有 P1 到达，所以 P1 必须先跑（哪怕它最长）。
第 2 步 · 画时间轴
• 9:00：P1 到达，CPU 空，P1 开始跑。
• 9:10：P2 到达，但 P1 在跑，P2 等待。
• 9:15：P3 到达，但 P1 在跑，P3 等待。
• 10:00：P1 跑完（60 分钟）。此时等待队列里有 P2（需 45 分）和 P3（需 25 分）。SJF 选短的，所以 P3 先跑。
第 3 步 · 填表
进程到达运行开始完成周转时间
P19:00609:0010:0010:00−9:00=60
P39:152510:0010:2510:25−9:15=70
P29:104510:2511:1011:10−9:10=120
例子注意：完成顺序是 P1 → P3 → P2，不是到达顺序 P1→P2→P3。
第 4 步 · 算平均周转时间
平均周转时间 T = (60 + 70 + 120) / 3 = 250 / 3 ≈ 83.3 分钟
答：完成顺序 P1→P3→P2；平均周转时间 ≈ 83 分 20 秒。

演示 2 · 生产者-消费者 PV 代码（逐行注释）

这是另一道大题的标准模板。背熟这个模板，所有 PV 题都是它的变形。

场景：生产者往缓冲区放产品，消费者从缓冲区取产品。缓冲区容量 N。
要求：写出生产者和消费者的 PV 同步代码。
第 1 步 · 设三个信号量（这是套路，记住）
• empty = N　（空位数，初值 N，表示一开始 N 个位置全空）
• full = 0　　（产品数，初值 0，表示一开始没产品）
• mutex = 1（互斥锁，初值 1，保证一次只有一个人操作缓冲区）
第 2 步 · 写生产者代码
Producer: 生产一个产品; P(empty); // 申请一个空位（空位-1），没有就等 P(mutex); // 锁住缓冲区（互斥） 把产品放入缓冲区; V(mutex); // 解锁缓冲区 V(full); // 产品数+1，唤醒可能在等的消费者
关键P(empty) 必须在 P(mutex) 前面！如果反过来：先锁住缓冲区，再发现没空位，就永远等不到 V(empty)，死锁。
第 3 步 · 写消费者代码（和生产者对称）
Consumer: P(full); // 申请一个产品（产品-1），没有就等 P(mutex); // 锁住缓冲区 从缓冲区取出产品; V(mutex); // 解锁 V(empty); // 空位+1，唤醒可能在等的生产者 消费这个产品;
第 4 步 · 记忆口诀
生产者：先 P 空、再 P 锁 → 放 → V 锁、V 满
消费者：先 P 满、再 P 锁 → 取 → V 锁、V 空
规律：P 同步在前，P 互斥在后；V 的顺序无所谓。
这就是 PV 题的万能模板：换个场景（司机售票员、理发师、吃水果），改的只是信号量的名字和含义，骨架完全一样。

演示 3 · 银行家算法安全性检查（完整推演）

死锁大题的标准题型：给矩阵，让你判断系统是否安全，找安全序列。

系统有 3 个进程 P1/P2/P3，共 12 台磁带机。
T0 时刻状态：
　P1：最大需求 10，已分 5
　P2：最大需求 4，已分 2
　P3：最大需求 9，已分 2
问：系统是否安全？给出安全序列。
第 1 步 · 算 Need（还需要的资源数）
Need = Max − Allocation
• P1：Need = 10 − 5 = 5
• P2：Need = 4 − 2 = 2
• P3：Need = 9 − 2 = 7
第 2 步 · 算当前可用 Available
总共 12 台，已分 5+2+2 = 9 台
Available = 12 − 9 = 3 台
第 3 步 · 安全性检查：逐个找"Need ≤ Work"的进程
设 Work = Available = 3，Finish[i] = false（全部）。

第 1 轮扫描：找 Need ≤ 3 的进程
• P1：Need 5 > 3　✗ 不满足
• P2：Need 2 ≤ 3　✓ 满足！让 P2 跑完，它归还已分的 2 台。
　　Work = 3 + 2 = 5；Finish[P2] = true。
• P3：Need 7 > 3　✗ 不满足（用当前 Work=5 比也不满足）

第 2 轮扫描（Work 现在是 5）：
• P1：Need 5 ≤ 5　✓ 满足！P1 跑完归还 5 台。
　　Work = 5 + 5 = 10；Finish[P1] = true。

第 3 轮扫描（Work 现在是 10）：
• P3：Need 7 ≤ 10　✓ 满足！P3 跑完归还 2 台。
　　Work = 10 + 2 = 12；Finish[P3] = true。
第 4 步 · 检查所有 Finish
Finish[P1] = Finish[P2] = Finish[P3] = true
全部为 true → 系统安全。
答：系统安全，安全序列为 <P2, P1, P3>。
（注意：每一步归还的是该进程的 Allocation，不是 Need）

演示 4 · 银行家算法「请求判定」（真题高频考法）

演示 3 只做了 T0 安全性检查。但真题常考后半步：某进程提出请求后，能否分配？这要"试探分配 + 重新检查安全性"。

接演示 3 的系统（3 进程，12 磁带机，T0 安全，Available=3）。
现在 P1 提出请求 Request = 2 台。
问：能否满足 P1 的请求？
第 1 步 · 请求判定三步审查
设 Request₁ = 2（P1 请求 2 台），逐项检查：
① Request ≤ Need?　P1 的 Need = 5，2 ≤ 5 ✓
② Request ≤ Available?　Available = 3，2 ≤ 3 ✓
③ 两关都过，进入"试探分配"。
第 2 步 · 试探分配（修改数据结构）
假装把资源分给 P1，更新：
• Available' = 3 − 2 = 1
• Allocation₁' = 5 + 2 = 7（P1 现在手上有 7 台）
• Need₁' = 5 − 2 = 3（P1 还需 3 台）
注意这一步只是"假装分"，不是真分。如果下一步安全性检查不通过，要恢复原数据。
第 3 步 · 试探分配后重新做安全性检查
新 Work 初值 = Available' = 1：
第 1 轮（Work=1）：
• P1：Need' 3 > 1 ✗
• P2：Need 2 > 1 ✗
• P3：Need 7 > 1 ✗
没有任何进程 Need ≤ 1 → 找不到安全序列 → 试探分配后系统不安全！
第 4 步 · 判定与恢复
安全性检查不通过 → 不能满足 P1 的请求，P1 必须等待。
恢复试探前的数据：Available 回到 3，Allocation₁ 回到 5，Need₁ 回到 5。
结论请求判定 = 三步审查（Request≤Need、Request≤Available、试探后重判安全），任一不满足就拒绝/等待。
答：P1 请求 2 台不能满足。试探分配后 Available 仅剩 1 台，无任何进程能运行，系统不安全，P1 必须等待。
核心：请求判定 = 三步审查，第三步"试探后重判安全性"是最容易漏的，一定要做。

09误解与边界

误解 1：就绪可以直接到等待
易错三态转换有严格方向：就绪→运行、运行→就绪、运行→等待、等待→就绪。没有就绪→等待，也没有等待→运行。
原文：状态变迁是有方向的。
如何避免：记口诀"就绪只去运行，等待只回就绪"。
误解 2：P 操作就是减 1，V 操作就是加 1
推论错简化记忆容易漏掉关键：P 减完后若 S<0，当前进程会被阻塞（不是继续跑）；V 加完后若 S≤0，要唤醒一个等待进程。
原文：「P 操作 S−1；若结果 <0 则阻塞自己入等待队列」「V 操作 S+1；若结果 ≤0 则唤醒一个等待进程」
关键：P 有阻塞效果，V 有唤醒效果。
误解 3：不安全状态 = 死锁
大题扣分点不安全只是"找不到安全序列"，但实际进程未必按最坏情况申请，所以可能不死锁。
正确关系：死锁 → 不安全；不安全 ⇏ 死锁；安全 → 不死锁。
原文：「并非所有不安全状态都是死锁状态……反之只要系统处于安全状态，系统便可避免进入死锁状态。」
误解 4：银行家算法 = 死锁预防
事实错两者都"防止死锁"，但方式不同：死锁预防=静态破坏四条件之一；死锁避免=运行时用银行家动态检查安全性。银行家属于避免，不是预防。
口诀：预防破坏条件，避免检查状态。

10背景知识缺口

读完本章如果还卡，通常缺的是下面这些（现在可以不全补，知道缺口在哪即可）：

  • 中断与时钟：时间片是怎么实现的？答：时钟硬件每隔一段时间发中断，OS 借机重新调度。这是运行→就绪的硬件基础。
  • 原语：PV 操作为什么不能被打断？答：它们是原语（不可分割的基本操作），由 OS 内核保证原子性。
  • 资源分配图（RAG）：死锁检测的图形化方法。本章只考银行家（避免），检测（RAG 化简）了解即可。
  • 多级反馈队列：六大调度算法里最复杂的一个，考试较少单独出大题，理解"优先级递减+时间片加倍"即可。
